# 项目结构总览

本文面向第一次接手这个仓库的开发者，目标是不读源码也能先建立整体认知：项目做什么、怎么运行、代码分层如何组织、数据怎样流转、为什么同时存在 Python 主应用和一个独立的 Three.js 渲染子项目。

文档基线时间为 **2026-04-05**。编写前 `md/` 目录为空，本文件是该目录下的第一篇项目总览。以下说明聚焦手写源码和运行资产；`node_modules/`、`dist/`、`__pycache__/` 等目录只说明角色，不逐项展开。

## 1. 项目定位与技术栈

这是一个用于求职展示的机器人软件产品原型，主题是“仿真机械臂任务验证平台 MVP”。它并不追求工业级控制，而是用较低的工程成本把一条完整的任务验证闭环产品化表达出来：

- 启动抓取并放置任务
- 实时查看状态、阶段、关键事件和动作视图
- 生成结构化结果报告
- 回放任务过程
- 对比不同策略版本
- 运行基准评测并汇总稳定性

主技术栈分成两层：

- **主应用层**：`Python + Streamlit + JSON`
- **动作渲染层**：`web/robot_renderer`，一个独立构建的 `Three.js + TypeScript + Vite` 子项目

可以把它理解成：

`Streamlit 负责产品界面和流程编排，Python 负责数据模型/模拟/存储，Three.js 子项目负责更强的动作展示。`

  Windows 下的一键启动脚本，会检查 Python、安装依赖并运行 `streamlit run app.py`。
- `requirements.txt`
  Python 依赖清单，目前核心依赖是 `streamlit`。

## 3. 先建立整体架构感

这个项目的核心链路可以概括成：

`Streamlit 页面 -> JsonStore -> simulator -> dataclass 模型 -> data/store.json -> renderer payload -> Three.js / SVG 渲染`

拆开看：

1. `app.py` 提供 5 个页面：
   `总览`、`任务详情`、`任务回放`、`版本对比`、`基准评测`。
2. 页面需要数据时，先通过 `JsonStore` 从 `data/store.json` 读出结构化对象。
3. 如果用户发起新任务或新评测，`JsonStore` 会调用 `simulator.py` 生成新的运行记录、事件和回放帧。
4. 这些数据会被映射为页面上的指标、时间线、摘要卡片，以及给渲染器使用的 `renderer payload`。
5. 如果 Three.js bundle 可用，则由 `web/robot_renderer/dist/renderer.js` 渲染 2.5D 场景。
6. 如果 bundle 不存在，或者浏览器侧渲染失败，页面会自动回退到 SVG 视图。

这也是为什么仓库里同时有 Python 和 TypeScript：

- Python 负责“业务真相”和“数据生成”
- TypeScript 负责“动作可视化体验”

## 4. 入口文件 `app.py` 在做什么

`app.py` 是最应该先读的文件，因为它把所有模块串起来了。

### 4.1 页面与导航

应用内固定有 5 个页面：

- 总览
- 任务详情
- 任务回放
- 版本对比
- 基准评测

页面切换和上下文主要放在 `st.session_state` 里，核心状态包括：

- 当前页面
- 当前角色视角
- 当前选中的运行记录
- 左右对比运行记录
- 当前选中的评测批次
- 当前动作渲染模式（`threejs` 或 `svg`）
- 启动任务时的预览签名、预览锚点和自动解析出的场景

### 4.2 五个页面各自负责什么

- **总览**
  展示项目介绍、启动任务表单、策略版本看板、最近任务列表。
- **任务详情**
  实时展示单次运行的状态、阶段、进度、关键事件、结果摘要和动作视图。
- **任务回放**
  用滑块选择回放帧，查看某个时刻的动作状态，并同步展示截至该时刻的事件。
- **版本对比**
  并排比较两次运行，输出成功差异、耗时差异、失败原因差异和推荐结论。
- **基准评测**
  发起一轮 benchmark，展示成功率、平均耗时、失败分布和每个用例的摘要。

### 4.3 `app.py` 和其他模块的关系

- 从 `robot_mvp.storage` 拿 `JsonStore`
- 从 `robot_mvp.simulator` 拿几乎全部业务计算函数
- 从 `robot_mvp.renderer` 把 `renderer payload` 送到 Three.js bundle
- 从 `robot_mvp.models` 使用 `RunEvent`、`ReplayFrame` 等类型

也就是说，`app.py` 主要负责“编排和展示”，不是业务规则本身的归属地。

## 5. 核心包 `robot_mvp/` 怎么分层

### 5.1 `models.py`：统一数据对象

这个文件定义了项目里的核心 dataclass，是整个系统的数据底座。最重要的类型有：

- `TaskTemplate`
  任务模板，描述任务名称、步骤和成功标准。
- `StrategyVersion`
  策略版本，描述不同策略的名称、版本号和说明。
- `BenchmarkSuite`
  评测套件，描述一组 benchmark case 和关注指标。
- `BenchmarkBatch`
  一次基准评测批次，记录批次时间、所用策略、套件和关联运行记录。
- `RunResult`
  单次运行的结果结论，包含成功与否、最终状态、失败原因、质量分、摘要说明。
- `RunRecord`
  单次运行的主记录，包含开始/结束时间、耗时、输入参数、标签和 `RunResult`。
- `RunEvent`
  运行过程中的关键事件，面向时间线展示。
- `ReplayFrame`
  回放帧，面向动作回放和 Three.js/SVG 渲染。

这些类型都有 `to_dict()` / `from_dict()`，用于和 `store.json` 互转。

### 5.2 `simulator.py`：业务核心

`simulator.py` 是整个项目最重要的业务模块，负责“生成数据”和“解释数据”。它大致做了 6 类事情。

#### 1. 定义演示世界

文件里内置了：

- 预设场景：`success`、`grasp_slip`、`placement_offset`
- 预设标签：`auto`、`标准成功`、`抓取失败`、`放置偏移`
- 节奏模式：默认 / 快节奏 / 慢速讲解
- 聚焦模式：全链路 / 抓取 / 放置 / 失败
- 策略版本、物料名、料仓、托盘位、benchmark case 定义

这些定义决定了“平台里会演什么”和“演出来是什么样”。

#### 2. 生成种子数据

`seed_store_data()` 会在首次运行时初始化默认数据，包括：

- 1 个任务模板
- 3 个策略版本
- 2 个 benchmark suite
- 一批示例运行记录、事件和回放帧

这就是为什么项目开箱后就能直接演示，而不是先接数据库或真机。

#### 3. 生成单次运行和评测批次

几个关键入口：

- `create_live_run_bundle()`
  生成一次“正在运行”的任务数据，用于真正点击“开始一次新验证”时写入 store。
- `create_preview_bundle()`
  生成启动表单右侧的动态预览，不会持久化到主数据文件。
- `create_benchmark_batch_bundle()`
  根据 benchmark suite 批量生成多条运行，并打包成一个 `BenchmarkBatch`。
- `create_run_bundle()`
  更底层的公共构造函数，负责把场景模板转成 `RunRecord + RunEvent + ReplayFrame`。

可以理解为：所有“演示用的运行数据”最终都由这里合成。

#### 4. 把备注解析成动态展示规则

`parse_operator_note_dynamic_profile()` 会从备注里识别关键词，例如：

- “快节奏”
- “慢速讲解”
- “强调抓取”
- “强调放置”
- “强调失败”

解析结果会写进运行的 `input_params["dynamic_profile"]`，后续影响：

- 页面里显示的节奏和聚焦标签
- 预览 / 实时视图的播放速度
- Three.js 场景的镜头和高亮重点

#### 5. 为页面提供业务计算

这一层是 `app.py` 最常调用的函数，例如：

- `project_run_view()`：把一条运行投影成“当前时刻应该看到的状态”
- `interpolate_replay_frame()`：对回放帧做时间插值
- `build_comparison_summary()`：生成对比摘要和推荐结果
- `summarize_benchmark_batch()`：生成 benchmark 结果摘要
- `generate_strategy_metrics()`：生成策略看板指标
- `iter_run_assets()`：按运行 ID 取回记录、事件和帧

#### 6. 生成前端渲染载荷

`build_renderer_payload()` 是 Python 到前端渲染器之间最重要的边界之一。

它输出的 payload 包含：

- 页面视图类型 `view_mode`
- 标题 `title`
- 运行状态、是否成功、当前阶段、进度
- 场景标签和解析后的场景
- 动态配置 `dynamic_profile`
- 物体/料仓/托盘等 `scene` 信息
- 动画模式、总时长、当前时间点
- 全量 `frames`
- 全量 `events`

Three.js 子项目并不直接理解 Python dataclass，而是消费这个 payload。

### 5.3 `storage.py`：JSON 持久化边界

`JsonStore` 是主应用和 `data/store.json` 之间的唯一主要边界，也是这套 MVP 的“简化数据层”。

它负责：

- `ensure_bootstrap()`：文件不存在时自动初始化
- `load()` / `save()`：读写 JSON
- `sync_running_runs()`：把到了结束时间的运行状态同步到最终状态
- `create_live_run()`：创建一条新运行并写入 `store.json`
- `create_benchmark_batch()`：创建 benchmark 批次并连带生成多条运行
- `get_run_bundle()`：按运行 ID 读取记录、事件和帧

如果以后要把 JSON 改成数据库，最直接的切入点就是替换这一层。

### 5.4 `renderer.py`：把 Python 接到前端 bundle

`renderer.py` 的职责很明确：

- 判断 `web/robot_renderer/dist/renderer.js` 是否存在
- 把 `build_renderer_payload()` 生成的 payload 序列化成 JSON
- 通过 `streamlit.components.v1.html` 注入页面
- 调用浏览器侧的 `window.RobotRenderer.render(root, payload)`
- 如果 Three.js bundle 不可用或运行失败，则保留 SVG 回退视图

这就是 Python 主应用和 Three.js 子项目之间的桥接层。

## 6. 单次运行的数据是怎么生成并展示的

以“开始一次新验证”为例，主链路如下：

1. 用户在总览页选择任务模板、策略版本、演示预设和备注。
2. `app.py` 先调用 `create_preview_bundle()` 做右侧动态预览。
3. 用户点击提交后，`JsonStore.create_live_run()` 被调用。
4. `create_live_run_bundle()` 根据策略、预设和时间，决定最终场景。
5. `create_run_bundle()` 生成：
   - `RunRecord`
   - `RunEvent[]`
   - `ReplayFrame[]`
6. `JsonStore` 把这些对象写进 `data/store.json`。
7. `任务详情` 页面再通过 `project_run_view()` 投影出当前时刻该显示的状态。
8. 页面把结构化数据转成指标卡、事件时间线、结果摘要和 `renderer payload`。
9. 动作视图由 Three.js 渲染；如果失败则回退到 SVG。

这个设计的优点是：**页面展示和数据来源是分离的**。页面不需要自己推断逻辑，只消费结构化数据。

## 7. 基准评测的数据是怎么生成的

基准评测不是“另一套系统”，而是建立在同一批数据对象之上的批量运行模式。

主链路如下：

1. 用户在 `基准评测` 页面选择任务模板、评测套件、策略版本。
2. `JsonStore.create_benchmark_batch()` 被调用。
3. `create_benchmark_batch_bundle()` 会遍历 suite 下的每个 case。
4. 每个 case 都会再次调用 `create_run_bundle()` 生成一条独立运行。
5. 多条运行的 ID 被收集进一个 `BenchmarkBatch`。
6. `summarize_benchmark_batch()` 再把这些运行汇总成：
   - 成功率
   - 平均耗时
   - 失败分布
   - 最弱案例
   - 推荐结论

因此，“单次运行”和“批量评测”本质上共享同一个数据模型，只是组织方式不同。

## 8. 前端子项目 `web/robot_renderer/` 为什么是独立的

这个子项目不是一个独立 Web 应用，而是一个**被主应用内嵌使用的渲染库**。

### 8.1 为什么拆出去

原因主要有三个：

- Three.js 场景、动画插值和镜头逻辑不适合直接写在 Python 字符串模板里
- 独立打包后，主应用只需要加载一个 `renderer.js`
- 可以单独做 TypeScript 类型约束和前端单测

### 8.2 `src/payload.ts`：定义前端理解的数据协议

这个文件负责：

- 定义 `RendererPayload`、`FramePayload`、`EventPayload` 等前端类型
- 归一化 `dynamic_profile`
- 计算动画时间 `computeAnimationTime()`
- 在帧之间采样 `sampleFrame()`
- 根据物体标签推断几何类型 `resolveObjectKind()`
- 查找失败发生的时间点 `findFailureOffset()`

可以把它理解成“前端版的渲染协议层”。

### 8.3 `src/index.ts`：Three.js 2.5D 场景本体

这个文件负责：

- 创建场景、相机、灯光、地面、料仓、托盘和机械臂部件
- 根据 payload 和当前时间计算采样状态
- 更新机械臂、夹爪、工件位置和镜头
- 根据 `highlight_mode` 决定抓取/放置/失败的聚焦方式
- 把渲染入口暴露为 `window.RobotRenderer = { render }`

因此主应用只要给它一个根节点和一个 payload，就能渲染出当前动作状态。

### 8.4 `vite.config.ts`：把渲染器打成主应用可加载的 bundle

这里的关键配置是：

- 入口：`src/index.ts`
- 输出：`dist/renderer.js`
- 格式：IIFE 库
- 名称：`RobotRenderer`

所以最终主应用在浏览器里拿到的是一个全局对象 `window.RobotRenderer`。

### 8.5 `tests/payload.test.ts`：前端侧的基础行为验证

目前前端测试重点不在重 UI，而在渲染协议和采样逻辑：

- `computeAnimationTime()` 是否正确处理循环动画和节奏倍速
- `sampleFrame()` 是否能正确插值
- `resolveObjectKind()` 是否能把中文物体标签映射到几何类型

这和 Python 测试形成互补：Python 保证业务数据正确，TypeScript 保证渲染协议正确。

## 9. `data/store.json` 里到底存了什么

`data/store.json` 是这个 MVP 的唯一主数据文件，目前承载的顶层结构包括：

- `task_templates`
- `strategy_versions`
- `benchmark_suites`
- `benchmark_batches`
- `run_records`
- `run_events`
- `replay_frames`

换句话说，这个文件不是只存“结果”，而是把运行前配置、运行中事件、运行后结论和回放数据都放在一起。这样做的好处是：

- 本地 demo 简单
- 不需要数据库
- 导出和备份方便
- 页面之间可以共享同一事实来源

代价也很明显：更适合单机 MVP，不适合多人协作或高并发场景。

## 10. 运行、构建与测试命令

### 10.1 主应用运行

安装依赖：

```powershell
python -m pip install -r requirements.txt
```

启动 Streamlit：

```powershell
streamlit run app.py
```

如果是在 Windows 环境，也可以直接运行：

```powershell
start.bat
```

`start.bat` 会先检查 Python，再执行依赖安装，最后启动 Streamlit。

### 10.2 前端渲染器构建

进入子项目目录后，常用命令是：

```powershell
npm run build
npm run check
npm test
```

其中：

- `npm run build` 会生成 `web/robot_renderer/dist/renderer.js`
- `npm run check` 运行 TypeScript 类型检查
- `npm test` 运行 Vitest 单测

### 10.3 当前已验证结果

截至本次文档整理时，已确认：

- Python `unittest`：**11 项通过**
- 前端渲染器 `vitest`：**3 项通过**

Python 测试主要覆盖：

- 种子数据是否完整
- 每条运行是否带事件和回放帧
- live run 是否会在未来时刻收敛到最终状态
- 对比摘要是否优先推荐成功运行
- 备注关键词是否能映射到动态配置
- 回放插值是否正确
- renderer payload 是否包含必须字段
- benchmark 汇总是否合理
- JSON 存储层是否能创建运行和评测批次

## 11. 哪些目录不用一开始深读

下面这些目录或文件先知道角色就够了：

- `web/robot_renderer/node_modules/`
  前端依赖安装目录，不是业务源码。
- `web/robot_renderer/dist/`
  前端构建产物目录，主应用只消费最终的 `renderer.js`。
- `__pycache__/`
  Python 字节码缓存。
- `tests/__pycache__/`
  测试运行时产生的缓存。
- `.tmp-tests/`
  Python 测试过程中使用的临时存储目录。

如果你在做功能开发，绝大多数时间应该集中在：

- `app.py`
- `robot_mvp/`
- `tests/test_simulator.py`
- `web/robot_renderer/src/`

## 12. 建议的阅读顺序

如果你要在最短时间内理解项目，推荐顺序如下：

1. 读 `README.md`
   先理解项目目标、运行方式和演示顺序。
2. 读 `app.py`
   建立页面、交互和主流程认知。
3. 读 `robot_mvp/models.py`
   先认清有哪些核心对象。
4. 读 `robot_mvp/simulator.py`
   再理解数据是如何生成、投影和汇总的。
5. 读 `robot_mvp/storage.py`
   明白数据什么时候被写入 `store.json`。
6. 读 `robot_mvp/renderer.py`
   看主应用如何把 payload 送到前端渲染器。
7. 读 `web/robot_renderer/src/payload.ts`
   了解前端如何理解 Python 发来的数据。
8. 读 `web/robot_renderer/src/index.ts`
   最后再看 Three.js 场景细节。

## 13. 一句话总结

这个仓库本质上是一个“用本地 JSON 驱动的机器人任务验证产品原型”：

- `app.py` 负责产品界面和流程
- `robot_mvp/` 负责业务逻辑、数据对象和存储
- `data/store.json` 负责承载运行事实
- `web/robot_renderer/` 负责把这些事实变成更有表现力的动作画面

如果你只想抓主干，先记住一句话：

**这是一个用 Streamlit 承载产品流程、用 Python 生成结构化运行数据、再用独立 Three.js 子项目增强动作展示的单机 MVP。**
